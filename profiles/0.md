# Profile 0 — System

The System profile is implemented by all conformant OwnTelemetry endpoints. It handles device lifecycle events (birth, will), server-initiated liveness probing (ping/pong), server-issued commands, and remote provisioning.

## Packet Types

| Type | Name              | Direction | Description                                      |
|------|-------------------|-----------|--------------------------------------------------|
| 1    | `birth`           | e->s      | Device online announcement                       |
| 2    | `will`            | e->s      | Graceful offline notification                    |
| 3    | `ping`            | s->e      | Server liveness probe                            |
| 4    | `pong`            | e->s      | Response to a server ping                        |
| 5    | `command`         | s->e      | Server-issued command                            |
| 6    | `command_ack`     | e->s      | Endpoint response to a command                   |
| 7    | `provision`       | s->e      | Server-initiated configuration update            |
| 8    | `provision_ack`   | e->s      | Endpoint response to a provision packet          |

---

## Type 1 — Birth

Sent when the endpoint comes online. Declares which telemetry profiles the device supports and its expected reporting interval — the server uses `interval` to determine when a device should be considered stale or offline.

| CBOR Key | Field Name | Type         | Optional | Description                                                                  |
|----------|------------|--------------|----------|------------------------------------------------------------------------------|
| 0        | `profile`  | uint8        |          | Always `0` for this profile                                                  |
| 1        | `id`       | bytes (8)    |          | Endpoint device identifier                                                   |
| 2        | `mid`      | uint8        |          | Rolling message ID                                                           |
| 3        | `ack`      | bool         |          | `true` if endpoint requests acknowledgment                                   |
| 4        | `type`     | uint8        |          | Always `1` for birth                                                         |
| 5        | `profiles` | array[uint8] |          | List of telemetry profile numbers this device supports                       |
| 6        | `firmware` | string       | yes      | Firmware version string (e.g. `"1.2.3"`)                                    |
| 7        | `interval` | uint16       |          | Expected seconds between telemetry reports; `0` = event-driven or irregular |

### Example

```json
{
  "profile": 0,
  "id": "a3f1c2b4d5e6f7a8",
  "mid": 1,
  "ack": true,
  "type": 1,
  "profiles": [2],
  "firmware": "2.1.0",
  "interval": 30
}
```

## Type 2 — Will

Sent when the endpoint is about to go offline gracefully. Servers should mark the device offline on receipt. If no will is received and the device exceeds its `interval` threshold (from the birth message) without sending a packet, the server may independently mark it as stale.

| CBOR Key | Field Name  | Type      | Description                                |
|----------|-------------|-----------|--------------------------------------------|
| 0        | `profile`   | uint8     | Always `0` for this profile                |
| 1        | `id`        | bytes (8) | Endpoint device identifier                 |
| 2        | `mid`       | uint8     | Rolling message ID                         |
| 3        | `ack`       | bool      | `true` if endpoint requests acknowledgment |
| 4        | `type`      | uint8     | Always `2` for will                        |
| 5        | `reason`    | uint8     | Reason code (see above)                    |
| 6        | `timestamp` | uint32    | Unix timestamp in seconds                  |

### Example

```json
{
  "profile": 0,
  "id": "a3f1c2b4d5e6f7a8",
  "mid": 88,
  "ack": false,
  "type": 2,
  "reason": 2,
  "timestamp": 1622634200
}
```

## Type 3 — Ping

Sent by the server to check whether an endpoint is reachable and alive. Uses the server->endpoint packet structure (`target_id` instead of `id`; no `ack` field). The server's `mid` is echoed back in the pong.

| CBOR Key | Field Name  | Type      | Description                               |
|----------|-------------|-----------|-------------------------------------------|
| 0        | `profile`   | uint8     | Always `0` for this profile               |
| 1        | `target_id` | bytes (8) | `id` of the endpoint being probed         |
| 2        | `mid`       | uint8     | Server's rolling message ID for this ping |
| 3        | `type`      | uint8     | Always `3` for ping                       |

### Example

```json
{
  "profile": 0,
  "target_id": "a3f1c2b4d5e6f7a8",
  "mid": 5,
  "type": 3
}
```

---

## Type 4 — Pong

Sent by the endpoint in response to a ping. `ping_mid` references the server's ping so the server can correlate response to probe and measure round-trip time.

| CBOR Key | Field Name | Type      | Description                                     |
|----------|------------|-----------|-------------------------------------------------|
| 0        | `profile`  | uint8     | Always `0` for this profile                     |
| 1        | `id`       | bytes (8) | Endpoint device identifier                      |
| 2        | `mid`      | uint8     | Rolling message ID                              |
| 3        | `ack`      | bool      | Typically `false` — pongs are not acked         |
| 4        | `type`     | uint8     | Always `4` for pong                             |
| 5        | `ping_mid` | uint8     | `mid` from the server's ping being responded to |

### Example

```json
{
  "profile": 0,
  "id": "a3f1c2b4d5e6f7a8",
  "mid": 89,
  "ack": false,
  "type": 4,
  "ping_mid": 5
}
```

| 2    | `reboot`        | none                             | Request a device reboot                   |

| Code | Name              | Description                                |

## Type 5 — Command

Sent by the server to issue a command to an endpoint. Uses the server->endpoint packet structure (`target_id` instead of `id`; no `ack` field). `command_data` is optional — its format and interpretation depend on `command_code`.

| CBOR Key | Field Name     | Type      | Optional | Description                                          |
|----------|----------------|-----------|----------|------------------------------------------------------|
| 0        | `profile`      | uint8     |          | Always `0` for this profile                          |
| 1        | `target_id`    | bytes (8) |          | `id` of the endpoint being commanded                 |
| 2        | `mid`          | uint8     |          | Server's rolling message ID for this command         |
| 3        | `type`         | uint8     |          | Always `5` for command                               |
| 4        | `command_code` | uint8     |          | Command to execute (see above)                       |
| 5        | `command_data` | bytes     | yes      | Command parameters; format depends on `command_code` |

### Example — set_interval

`command_data` contains a single uint16 encoded as 2 bytes (big-endian): the new interval in seconds.

```json
{
  "profile": 0,
  "target_id": "a3f1c2b4d5e6f7a8",
  "mid": 12,
  "type": 5,
  "command_code": 1,
  "command_data": "001e"
}
```

### Example — reboot

```json
{
  "profile": 0,
  "target_id": "a3f1c2b4d5e6f7a8",
  "mid": 13,
  "type": 5,
  "command_code": 2
}
```

## Type 6 — Command Ack

Sent by the endpoint in response to a command. `command_mid` references the server's command `mid`. `result` is optional and command-specific.

| CBOR Key | Field Name    | Type      | Optional | Description                                        |
|----------|---------------|-----------|----------|----------------------------------------------------|
| 0        | `profile`     | uint8     |          | Always `0` for this profile                        |
| 1        | `id`          | bytes (8) |          | Endpoint device identifier                         |
| 2        | `mid`         | uint8     |          | Rolling message ID                                 |
| 3        | `ack`         | bool      |          | Typically `false`                                  |
| 4        | `type`        | uint8     |          | Always `6` for command_ack                         |
| 5        | `command_mid` | uint8     |          | `mid` from the server's command being responded to |
| 6        | `status`      | uint8     |          | Status code (see above)                            |
| 7        | `result`      | bytes     | yes      | Command-specific response data                     |

### Example

```json
{
  "profile": 0,
  "id": "a3f1c2b4d5e6f7a8",
  "mid": 90,
  "ack": false,
  "type": 6,
  "command_mid": 12,
  "status": 1
}
```

## Type 7 — Provision

Sent by the server to update persistent endpoint configuration. Uses the server->endpoint packet structure (`target_id` instead of `id`; no `ack` field). The provision packet is encrypted with the endpoint's **current** key.

| CBOR Key | Field Name       | Type      | Description                                          |
|----------|------------------|-----------|------------------------------------------------------|
| 0        | `profile`        | uint8     | Always `0` for this profile                          |
| 1        | `target_id`      | bytes (8) | `id` of the endpoint being provisioned               |
| 2        | `mid`            | uint8     | Server's rolling message ID for this provision       |
| 3        | `type`           | uint8     | Always `7` for provision                             |
| 4        | `provision_code` | uint8     | Provision operation (see above)                      |
| 5        | `data`           | bytes     | Provision payload; format depends on `provision_code` |

### Example — set_key

`data` contains the new 16-byte encryption key.

```json
{
  "profile": 0,
  "target_id": "a3f1c2b4d5e6f7a8",
  "mid": 20,
  "type": 7,
  "provision_code": 1,
  "data": "a3f1c2b4d5e6f7a81234567890abcdef"
}
```

---

## Type 8 — Provision Ack

Sent by the endpoint in response to a provision packet. `provision_mid` references the server's provision `mid`. The endpoint sets `ack` to `true` — the server's subsequent acknowledgment is the commit confirmation.

For `set_key` provisions, this packet is **encrypted with the new key** — it serves as the end-to-end test that the key is valid. The server attempts to decrypt it with the new key; success means the key works in both directions.

| CBOR Key | Field Name      | Type      | Description                                           |
|----------|-----------------|-----------|-------------------------------------------------------|
| 0        | `profile`       | uint8     | Always `0` for this profile                           |
| 1        | `id`            | bytes (8) | Endpoint device identifier                            |
| 2        | `mid`           | uint8     | Rolling message ID                                    |
| 3        | `ack`           | bool      | Always `true` — server ack is the commit confirmation |
| 4        | `type`          | uint8     | Always `8` for provision_ack                          |
| 5        | `provision_mid` | uint8     | `mid` from the server's provision packet              |
| 6        | `status`        | uint8     | Status code (see above)                               |

### Key Change Flow

1. Server sends `provision` (type 7) encrypted with the endpoint's **current** key; `data` contains the new key.
2. Endpoint decrypts with current key, stores new key as pending.
3. Endpoint sends `provision_ack` (type 8) encrypted with the **new key**; `ack = true`.
4. Server attempts to decrypt the `provision_ack` with the new key:
   - **Success** — server commits new key to its registry; sends standard ack encrypted with new key.
   - **Failure** — server cannot decrypt; it retains the old key and may retry.
5. Endpoint receives the server ack (decryptable with new key) → commits new key permanently.
6. If the endpoint does not receive an ack within its timeout, it reverts to the old key.

**Server grace period**: after sending the commit ack, the server retains the old key for a short transition window. If it receives a subsequent packet encrypted with the old key, the endpoint reverted (ack was lost) — the server rolls back its key record and can retry.

### Example

```json
{
  "profile": 0,
  "id": "a3f1c2b4d5e6f7a8",
  "mid": 91,
  "ack": true,
  "type": 8,
  "provision_mid": 20,
  "status": 1
}
```
