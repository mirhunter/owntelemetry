# Profile 10 ŌĆö Doorbell

Event-driven profile for doorbell devices with optional camera support. Unlike continuous-reporting profiles, this profile sends no periodic data ŌĆö packets are transmitted only when the button is pressed. There are no birth or will packets; the server treats absence of traffic as normal.

If the device has a camera, a snapshot is captured on button press and transmitted as a sequence of `image_chunk` packets immediately after the `button_event`. The `camera` field in the button event tells the server whether to expect chunks. The server reassembles the image using `button_mid`, `seq`, and `total`.

## Packet Types

| Type | Name            | Direction | Description                                      |
|------|-----------------|-----------|--------------------------------------------------|
| 1    | `button_event`  | eŌåÆs       | Button press notification                        |
| 2    | `image_chunk`   | eŌåÆs       | One chunk of a camera snapshot                   |

## Image Format Codes

Used in the optional `format` field of `image_chunk`. Only needs to appear in the first chunk (`seq = 0`) ŌĆö subsequent chunks can omit it.

| Code | Name   |
|------|--------|
| 1    | `jpeg` |
| 2    | `png`  |
| 3    | `bmp`  |

## Type 1 ŌĆö button_event

Sent immediately when the button is pressed. If `camera` is `true`, the server should expect `image_chunk` packets to follow with `button_mid` matching this packet's `mid`.

| CBOR Key | Field       | Type     | Description                                                    |
|----------|-------------|----------|----------------------------------------------------------------|
| 0        | `profile`   | uint8    | Always `10` for this profile                                   |
| 1        | `id`        | bytes(8) | Endpoint device identifier                                     |
| 2        | `mid`       | uint8    | Rolling message ID                                             |
| 3        | `ack`       | bool     | `true` if endpoint requests acknowledgment                     |
| 4        | `type`      | uint8    | Always `1` for button_event                                    |
| 5        | `timestamp` | uint32   | Unix timestamp of the button press                             |
| 6        | `camera`    | bool     | `true` if image chunks will follow; `false` for button-only   |

### Example

```json
{
  "profile": 10,
  "id": "a1b2c3d4e5f6a7b8",
  "mid": 12,
  "ack": true,
  "type": 1,
  "timestamp": 1718000000,
  "camera": true
}
```

## Type 2 ŌĆö image_chunk

One chunk of a camera snapshot. The image is split into fixed-size chunks by the endpoint and reassembled in order by the server. All chunks for a single snapshot share the same `button_mid` and `timestamp`.

| CBOR Key | Field        | Type     | Optional | Description                                                      |
|----------|--------------|----------|----------|------------------------------------------------------------------|
| 0        | `profile`    | uint8    |          | Always `10` for this profile                                     |
| 1        | `id`         | bytes(8) |          | Endpoint device identifier                                       |
| 2        | `mid`        | uint8    |          | Rolling message ID                                               |
| 3        | `ack`        | bool     |          | `true` if endpoint requests acknowledgment                       |
| 4        | `type`       | uint8    |          | Always `2` for image_chunk                                       |
| 5        | `timestamp`  | uint32   |          | Unix timestamp of the image capture (same across all chunks)     |
| 6        | `button_mid` | uint8    |          | `mid` of the `button_event` packet that triggered this capture   |
| 7        | `seq`        | uint16   |          | Chunk sequence number, 0-based                                   |
| 8        | `total`      | uint16   |          | Total number of chunks for this image                            |
| 9        | `data`       | bytes    |          | Raw chunk bytes                                                  |
| 10       | `format`     | uint8    | yes      | Image format code (see above); recommended on first chunk only   |

### Reassembly

The server collects chunks keyed by `(id, button_mid)` and orders them by `seq`. The image is complete when `total` chunks have been received. Out-of-order and duplicate delivery (e.g. over QoS 0) are handled by tracking received sequence numbers rather than a running count.

### Example ŌĆö first chunk of 8

```json
{
  "profile": 10,
  "id": "a1b2c3d4e5f6a7b8",
  "mid": 13,
  "ack": false,
  "type": 2,
  "timestamp": 1718000001,
  "button_mid": 12,
  "seq": 0,
  "total": 8,
  "data": "<binary>",
  "format": 1
}
```
